# UX Designer Review — Post-Overhaul Consistency Pass
**Reviewer**: Senior UX Designer (OmniProd perspective)
**Date**: 2026-04-03
**Scope**: Alerts page (primary), cross-page consistency with Patches, CVEs, Deployments, Settings

---

## Verdict: FAIL

The overhaul landed a lot of good structural work — stat cards, filter bars, and view toggles are now present across pages. But execution inconsistency between pages, multiple data display defects, and several interaction gaps prevent a PASS verdict. Approximately 30% of findings are major or critical; they will be noticed immediately by any evaluator.

---

## Findings

| ID | Severity | Element | Observation | Suggestion |
|----|----------|---------|-------------|------------|
| UX-01 | **critical** | Alerts — stat cards vs filter mismatch | Stat cards show "82 Total / 49 Critical / 31 Warning / 2 Info" but the table header reads "33 alerts" when the Active filter is selected. The numbers on the stat cards do not match the filtered view — they appear to be unfiltered totals while the table shows filtered results. A buyer will read this as a data integrity bug in the first 10 seconds. | Stat cards must either (a) always show unfiltered grand totals and be clearly labeled as such, or (b) update to reflect the active filter. If (a), add a subtle label such as "all time" or "total". If (b), update the query. Pick one and apply it consistently across Patches, CVEs, Deployments, and Alerts. |
| UX-02 | **critical** | Alerts card view — "Command timed out" repeated for all 33 rows | Every single visible alert has the same title ("Command timed out") and the same description text. This strongly suggests the alert title column is not being rendered — it is either hardcoded or the title field is mapping to the wrong API field. The card view makes this starkly visible because titles are the primary piece of information on each card. | Check the `title` field mapping in the alert hook and card component. Verify the API is returning distinct titles per alert. The table view also shows "Comman…" truncated in the TITLE column for all rows, which confirms the underlying data issue. |
| UX-03 | **critical** | Alerts table — TITLE column truncated so aggressively it conveys zero information | The TITLE column shows "Comman…" for every row without exception. At this column width, the title is effectively invisible. Even with distinct titles, this truncation level fails the information density standard. | Widen the TITLE column significantly — it is the most semantically important column on the table. As a minimum, show at least 30–40 characters. Ensure a tooltip on hover shows the full title (the long text standard requires this). |
| UX-04 | **critical** | Alerts — RESOURCE column shows raw internal paths | The RESOURCE column shows values like `command/88a022fa…`, `command/aadebb…`, `command/7762bf…` — these are raw internal entity paths/IDs, not human-readable resource names. This is a direct violation of the enterprise readiness standard: "UUIDs/internal IDs must never appear in the primary display of an entity." | Map resource type + ID to a readable label. If the resource is an agent command, show "Agent Command" or the deployment name. At minimum, strip the UUID portion and show only the resource type with the ID in a monospace chip or tooltip. |
| UX-05 | **major** | Stat card pattern is inconsistent across pages | Patches: 5 stat cards using a specific accent-border on the active card (green left border + white text). CVEs: 4 stat cards, no count on the Total card, different text sizing. Deployments: 4 stat cards with a distinct page-level heading "Deployments 27" above the cards (no equivalent on Alerts or Patches). Alerts: 4 stat cards. Every page uses a slightly different internal layout: some cards have the label below the number, some vary the number font size. | Define one canonical stat card anatomy: number (large, semibold) → label (small, muted, uppercase). Apply identically across all pages. Decide whether a page title with count badge goes above the stats or not, and apply that decision uniformly. |
| UX-06 | **major** | Deployments page has a redundant page-level heading with badge | The Deployments page shows a large "Deployments" `h1` plus a "27" count badge above the stat cards — no other page has this pattern. On Patches, CVEs, and Alerts, the top-bar simply says the page name and the stat cards provide the counts. This creates two different heading patterns within the same product. | Remove the inline count badge from the Deployments `h1` and let the stat cards carry that information, consistent with every other page. |
| UX-07 | **major** | Filter bar layout is inconsistent across pages | Patches: severity quick-filters and search on line 1, dropdown filters (OS Family, Status) on line 2, with "Create Deployment" CTA on line 2 right. CVEs: search and severity filters on line 1, toggle-style buttons (Exploit Available, KEV Only) plus dropdowns all on line 2. Deployments: search and status quick-filters on one line (overflowing off-screen — "mm/dd" date filter is cut off). Alerts: status quick-filters and search on line 1, category pills and date/limit controls on line 2, view-toggle on its own third line. | Standardize the filter bar to two lines max across all pages: Line 1 = search + key quick-filters. Line 2 = secondary filters + view toggle (right-aligned). CTA button should be consistently placed (right side of line 2 or in the page header). |
| UX-08 | **major** | Deployments filter bar overflows — date filter is cut off | On the Deployments filter bar, the date filter control ("mm/dd…") is visibly cropped at the right edge of the viewport. There is no horizontal scroll affordance. A filter control that cannot be reached is a broken filter. | Fix the filter bar layout to wrap or fit within the container. Use flex-wrap or reduce the number of visible items. |
| UX-09 | **major** | Patches card view — cards lack visual hierarchy | Patch cards show: title (CVE ID), a severity badge, and then three small data labels (CVES / CVSS / AFFECTED) with values of 0, 0.0, and 0. The entire card is monochromatic — there is no clear primary vs secondary information hierarchy. The action menu (⋮) is the only interactive affordance and it is very small. There are no action buttons on the card (unlike CVE cards which have "Detail" and "Deploy" buttons). | Add a primary CTA button to patch cards consistent with CVE cards. Give the patch name more visual weight (larger or bolder). Ensure the severity badge provides color differentiation. |
| UX-10 | **major** | CVE card view — CVSS ring gauge uses color alone | The CVSS score ring gauge changes color (gray for low, amber for medium, orange/red for high) but there is no text label explaining the CVSS category. Color-blind users cannot distinguish between score categories. | Add a text label inside or below the ring ("HIGH", "MEDIUM") alongside the numeric score. The score number alone is not self-sufficient without the category context. |
| UX-11 | **major** | Alerts — view toggle is orphaned on its own row | The list/card toggle (≡ ⊞) on the Alerts page sits alone on a third line within the filter bar area, separated from everything else with no clear relationship to adjacent controls. On Patches, CVEs and Deployments the toggle is correctly positioned at the right end of the filter bar on the same line as search/dropdowns. | Move the view toggle to the right end of the second filter row, consistent with other pages. This is the same pattern used everywhere else. |
| UX-12 | **major** | Alerts card — "Active" status label is green, same color as successful states | The "Active" status badge in alert cards is rendered in bright green — the same semantic color used for "Completed" deployments, "Healthy" endpoints, and passing compliance checks. An "Active" alert is not a positive state; it is an unresolved problem. | Change the "Active" alert status color to amber (warning) or blue (neutral/informational). Reserve green strictly for resolved/positive states. This is a semantic color consistency violation. |
| UX-13 | **major** | Alerts — STATUS column shows "ACTIVE" in bright green for unresolved alerts | Same semantic color issue as UX-12, but in the table view. The STATUS column renders "ACTIVE" in bright green for unresolved alerts requiring attention — green signals "all good" to the user. | Apply the same fix as UX-12 across both views. |
| UX-14 | **minor** | Patches — "Create Deployment" CTA is inside the filter bar container | The "Create Deployment" primary action button is placed inside the filter/toolbar box on line 2 next to the dropdown filters. On CVE card view, the CTAs (Detail/Deploy) are on the cards. On Deployments, there is no visible CTA. The inconsistent CTA placement means users cannot reliably find the primary action. | Place the primary CTA button in the page header area (top right), consistent with the standard pattern for list pages: `[Page Title] [...........] [Primary CTA]`. |
| UX-15 | **minor** | Deployments — no view toggle on the list view | The Deployments page does not show a list/card toggle button in the table view. The card view is accessible via `?view=card` URL parameter but there is no UI affordance to switch. Patches, CVEs, and Alerts all have the toggle visible. | Add the list/card toggle (≡ ⊞) to the Deployments filter bar, consistent with other pages. |
| UX-16 | **minor** | CVE stat card area is missing a "Total" card | The CVE page shows 4 stat cards: Critical (6), High (14), Medium (28), KEV Listed (11). There is no "Total" count card. Every other page — Patches (Total: 207,222), Deployments (Total: 27), Alerts (Total: 82) — leads with a Total stat card. | Add a Total CVE count card as the first card, matching the pattern of all other pages. |
| UX-17 | **minor** | Patches — stat card numbers have no thousands separators | The Patches stat cards show "207222", "30706", "79657", "92484" — raw integers with no comma separators. The data-integrity standard is explicit: "Any number above 999 without a separator is a violation." | Format all stat card numbers with locale-appropriate separators: 207,222 / 30,706 / 79,657 / 92,484. |
| UX-18 | **minor** | Alerts — CVSS score shows "0.0" across all rows in Patches table | The CVSS column shows "0.0" for every patch row. This is likely a null/zero conflation issue — newly ingested patches without CVSS scores should show "—" (em dash) not "0.0", which falsely implies a measured score of zero. | Treat CVSS = 0.0 as null/unmeasured and render "—" instead. Only display a numeric score when one has been explicitly calculated and returned from NVD. |
| UX-19 | **minor** | Settings sidebar — form labels use ALL CAPS monospace style | The Settings / General page renders form field labels ("ORGANIZATION NAME", "TIMEZONE", "DATE FORMAT", "DEFAULT SCAN INTERVAL") in small uppercase monospace. This is a very technical aesthetic that feels out of place for end-user settings. The same uppercase monospace style is used for table column headers — applying it to form labels creates confusion between data display contexts and input form contexts. | Use sentence-case labels for form fields ("Organization name", "Timezone") in a regular (non-monospace) sans-serif. Reserve the uppercase monospace style strictly for table column headers. |
| UX-20 | **minor** | Settings sidebar — "Back" link has no destination context | The Settings page shows a "← Back" link at the top of the sidebar with no label indicating where it returns to. Users who navigated from different pages would not know the destination. | Label it "← Back to App" or use a breadcrumb: "PatchIQ / Settings / General". This provides context regardless of navigation history. |
| UX-21 | **minor** | Alerts — "33 alerts" label below filter bar has no visual hierarchy | The text "33 alerts" appears as a plain line of body text between the filter bar and the table. It is the correct concept (showing how many results match the current filter) but it has no visual weight and could easily be missed. | Give this label a subtle style treatment — slightly muted text, or incorporate it into the filter bar as "Showing 33 results". If filters are active, show "33 of 82 alerts (filtered)". |
| UX-22 | **minor** | Deployments card — "Rollback" CTA uses amber/orange style inconsistently | On the Deployments card view, completed deployments show a "Rollback" button styled with an amber/orange border. Failed deployments also show "Rollback" in the same style. The button hierarchy standard states destructive actions should use the destructive (red) variant. Rollback is a significant, potentially impactful action. | If Rollback is a destructive action (and it is — it reverses applied patches), use the destructive button style (red). If it is a recoverable secondary action, use a ghost/secondary style. Do not use amber, which semantically means "warning/caution". |
| UX-23 | **nitpick** | Alerts — "Rules" button uses a slider icon (Settings2) | The "Rules" button in the Alerts filter bar uses a `Settings2` (sliders) icon. The consistency standard flags this: a sliders icon is already used for filter controls generically. A different icon (such as a list-rules or shield icon) would more clearly indicate "alert rules configuration" vs "filter controls". | Consider using a `Bell` + `Settings` compound icon, or a dedicated rules icon, to distinguish the Rules action from filter affordances. |
| UX-24 | **nitpick** | Patches — "Export" button is present but visually disabled at all times | The Export button on the Patches page appears grayed out/muted at all times, even when there are 207,222 patches in the list. There is no tooltip explaining when it becomes active. Users will not know if Export is an unimplemented feature, a permission-gated feature, or a bug. | Either enable Export and make it functional, or show a tooltip on the disabled button: "Export is available after applying a filter to reduce result size" or "Requires admin permission". |
| UX-25 | **nitpick** | Alerts — card view checkbox is left-aligned but table view has it too | Both views have checkboxes for bulk selection, which is good. However, in the card view, the checkbox is in the top-left corner of each card with no visible bulk action toolbar appearing when items are selected. Check whether the bulk action affordance (toolbar with "Acknowledge All", "Dismiss All") actually appears on card selection in card view. | Verify bulk action toolbar appears in card view when one or more cards are checked. If not, either implement it or remove the checkboxes from card view. |

## Summary by Area

### Critical (4)
Data/content defects that will be immediately noticed and erode trust: the stat card vs table count mismatch (UX-01), the homogeneous "Command timed out" title across all alerts (UX-02), the aggressively truncated TITLE column (UX-03), and raw internal resource paths in the RESOURCE column (UX-04).

### Major (9)
Consistency and semantic defects that break the visual contract: inconsistent stat card patterns (UX-05), redundant deployment heading (UX-06), filter bar layout divergence (UX-07), Deployments filter overflow (UX-08), patch card weak hierarchy (UX-09), CVE ring gauge color-only encoding (UX-10), Alerts view toggle orphaned (UX-11), green color for unresolved alert status in both table and card views (UX-12 + UX-13).

### Minor (9)
Noticeable quality gaps that degrade perceived polish but do not block use.

### Nitpick (3)
Taste-level observations.

---

## Priority Fix Order

1. UX-02 (alert title mapping) — likely a data bug, highest perceived impact
2. UX-04 (raw resource IDs) — enterprise readiness violation, immediately visible
3. UX-01 (stat card count mismatch) — buyer trust issue
4. UX-12 / UX-13 (green for active alerts) — semantic color violation
5. UX-03 (title column width) — usability
6. UX-07 / UX-08 / UX-11 (filter bar consistency + overflow) — polish
7. UX-17 (number formatting) — quick win, one-line fix
8. UX-18 (CVSS null vs zero) — data display integrity
9. UX-05 / UX-06 / UX-16 (stat card and heading consistency) — cross-page consistency pass
